# Linux Development

This skill covers **how to write Linux desktop code** in this repo. For the _workflow_ of implementing a spec, see `implementing-a-spec`. For _what to build_, see the spec.

This is the **Linux desktop client (GTK)**. The web app is the **reference implementation** — read its realization of a spec for context before mirroring behavior here idiomatically.

## The Client layer is generated, not hand-rolled

The Linux app reaches the backend through a **typed client generated by Progenitor** from the project's OpenAPI document, with **reqwest** (on a Tokio runtime) as the transport. You never assemble requests by hand. **Diesel/SQLite** is the on-device store — a local-first cache and offline source, not a second backend. Identity flows through **Clerk**; the generated client attaches the Clerk token. See `apps/linux/CLAUDE.md` for the wrapper that exposes idiomatic async call sites on top of the generated client.

## Idioms (read these before writing code)

### Relm4 component model

A component is a `Model` + `Input`/`Output` message enums + an `update` function + a `view!` macro. The **`update` logic is the view-model analog and the primary spec target** — keep it separate from widget construction (the `view!` block).

```rust
// SPEC: vm.items.list
pub struct ItemsList {
    state: ListState,
}

#[derive(Debug)]
pub enum ListInput {
    Load,
    Loaded(Vec<Item>),
    Failed(String),
}

pub enum ListState {
    Idle,
    Loading,
    Loaded(Vec<Item>),
    Error(String),
}

impl ItemsList {
    fn update(&mut self, input: ListInput) {
        match input {
            ListInput::Load => self.state = ListState::Loading,
            ListInput::Loaded(items) => self.state = ListState::Loaded(items),
            ListInput::Failed(message) => self.state = ListState::Error(message),
        }
    }
}
```

- State is a plain enum — exhaustive `match` in the view.
- User actions and async results arrive as `Input` messages; the component reacts, it does not block.
- Cross-component communication uses `Output` messages, not shared mutable state.
- The reverse pointer lives on the **component/update logic**, not the widget tree.

### GTK 4 + Adwaita widgets

Build the UI from Adwaita widgets — `adw::ApplicationWindow`, `adw::HeaderBar`, `adw::PreferencesGroup`, the `adw::*Row` family — and follow the GNOME HIG. Don't fight GTK conventions or reinvent affordances that the toolkit already ships. Widget construction belongs in the `view!` macro; keep it free of business logic.

### Async with Tokio

Network and database work runs off the GTK main thread. Spawn it (e.g. `relm4::spawn` / a Tokio task) and route the result back into the component as an `Input` message — never touch widgets from a worker task, and never block the main loop on IO.

```rust
fn load(&self, sender: ComponentSender<Self>) {
    let client = self.client.clone();
    relm4::spawn(async move {
        let message = match client.list_items().await {
            Ok(items) => ListInput::Loaded(items),
            Err(error) => ListInput::Failed(error.to_string()),
        };
        sender.input(message);
    });
}
```

### Tests at the update/logic layer

Behavioral tests target `update` and the pure logic around it — not the rendered widgets. Use the Rust test convention from `specs/CONVENTIONS.md`: a `#[cfg(test)] mod` carrying the spec ID, with a `// [scenario.<id>]` comment above each `#[test]` (Rust test names can't hold dots or brackets, so the scenario sub-ID lives in the comment drift tooling greps).

```rust
// SPEC: vm.items.list
#[cfg(test)]
mod tests {
    use super::*;

    // [scenario.items.list.empty]
    #[test]
    fn shows_empty_state_when_no_items_exist() {
        let mut component = ItemsList { state: ListState::Idle };
        component.update(ListInput::Loaded(vec![]));
        assert!(matches!(component.state, ListState::Loaded(ref items) if items.is_empty()));
    }
}
```

### Error handling

Use `Result` everywhere IO can fail. Define domain error types with `thiserror`; use `anyhow` for top-level glue where a typed error buys nothing. **No `unwrap()` / `expect()` on fallible IO in shipped paths** — propagate with `?` and surface a user-observable error in the component state. `unwrap` is acceptable only on invariants that genuinely cannot fail (and a comment should say why).

## Verifying

- **`mise run -C apps/linux test`** (`cargo test`) — the update/logic layer. This is the agent's primary verification surface.
- **`cargo clippy` + `cargo fmt --check`** before declaring done. Treat Clippy warnings as failures unless you have a documented reason.
- **`cargo run`** for manual visual checks — needs a display (a Wayland or X11 session). GTK GUI automation is heavy and the agent often has **no display**: verify component logic via `cargo test`, and have a human run the app for the visual pass. Don't claim the UI works from a test run alone.

## Commit

Land focused, atomic commits as the work hits natural boundaries — typically per spec ID, per component + its tests, or per cohesive refactor. See `.claude/rules/commit-discipline.md`.

Linux-specific notes:

- **`Cargo.toml` / `Cargo.lock` bumps go alone.** Dependency changes belong in their own commit (`chore: bump <crate>`), not bundled with feature code.
- **Generated Progenitor output rides with the codegen commit** that produced it — never commit it out of sync with the OpenAPI document it was generated from.
- **Don't commit build outputs.** `apps/linux/target/` is gitignored.
